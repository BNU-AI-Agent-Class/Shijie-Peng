## 2. 产品定位

HCAI Research Map 是一个面向 HCAI / Human-AI Interaction / Human-Centered AI 研究者的研究方向追踪与分析系统。

它不是普通论文搜索引擎，也不是机构排行榜，而是一个持续追踪 HCAI 领域研究动态的网页系统。最终 HTML 原型中的页面品牌为 HCAI Research Radar，但产品主轴是 Research Map：按研究方向、研究问题、论文内容结构和方向关系组织论文。

产品每天定时更新，自动抓取开放学术数据源中的 HCAI 相关论文，提取并展示：

1. 新增 HCAI 论文；
2. 18 个一级研究方向的热度；
3. 142 个二级研究问题的增长；
4. 论文对应的研究问题、研究方法、应用场景、用户群体、AI 系统类型、评价指标和成果类型；
5. 方向之间的共现关系、方法借用关系和演化路径；
6. 方向趋势、方向对比和每周研究摘要。

高校 / 机构信息仍保留，但作为论文与方向分析的辅助维度，不再是一期开屏主线。

产品希望回答的问题：

1. HCAI 当前有哪些研究方向；
2. 每个方向下面具体在研究什么问题；
3. 不同研究方向之间有什么关系；
4. 某篇论文解决了什么研究问题；
5. 论文使用了什么研究方法、用户群体、应用场景和评价指标；
6. 哪些主题正在变热；
7. 哪些研究问题已经被充分覆盖；
8. 哪些方向可能存在研究空白。

## 3. 产品背景

HCAI、Human-AI Interaction、人智交互、人本 AI、AI UX、多 Agent 协作、可信 AI、解释性 AI、AI 辅助决策等方向正在快速增长。

相关论文分散在多个数据库、会议、预印本平台、实验室主页和项目网站中。用户如果想系统性理解一个研究方向，需要反复搜索：

1. OpenAlex；
2. Semantic Scholar；
3. arXiv；
4. Crossref；
5. ACM Digital Library；
6. IEEE；
7. Google Scholar；
8. 会议论文集；
9. 实验室主页；
10. 论文项目页。

现有工具通常只能回答有哪些论文、谁发表了论文、引用数是多少，但用户真正想知道的是：

1. 这个方向到底在研究什么；
2. 研究问题之间是什么关系；
3. 论文成果之间如何互相补充；
4. 某个主题适合从哪里切入；
5. 心理学、HCI、AI 产品研究者能从哪些方向进入 HCAI。

本产品希望把分散在全网的 HCAI 论文，整理成一个按研究方向、研究问题、研究内容和成果关系组织的研究地图。

## 4. 产品目标

### 4.1 第一期目标：方向论文雷达 MVP

第一期重点解决“把 HCAI 相关论文稳定收集、分类、展示出来”。目标是做出一个可每日更新、可筛选、可解释的 HCAI 论文与研究方向看板。

第一期必须支持：

1. 每日 0:00 北京时间更新；
2. 抓取最近 72 小时新增或更新论文；
3. 识别 HCAI 相关论文；
4. 给论文打 HCAI 相关性分数；
5. 将论文归入一级研究方向；
6. 提取二级研究问题；
7. 提取论文内容字段：研究方法、应用场景、用户群体、AI 系统类型、交互模式、评价指标、成果类型；
8. 展示首页、论文列表、研究方向列表、论文详情、方向详情 5 个用户端页面；
9. HCAI 分数 ≥ 70 进入主看板；
10. HCAI 分数 50–69 进入待审池；
11. 论文详情页展示分类理由和数据质量；
12. 方向详情页展示方向定义、别名、研究问题构成、高频内容、代表论文和相关方向。

第一期不追求复杂趋势分析，而是先让用户可以清楚看到：最近 HCAI 领域出现了哪些论文，它们分别研究什么方向、什么问题、什么方法和什么场景。

### 4.2 第二期目标：研究方向路线分析

第二期以最终原型中的 Phase II 页面为准，包括：

1. 方向关系图；
2. 研究趋势页；
3. 方向对比页；
4. 每周摘要页；
5. 子方向画像 / 方向画像页；
6. Markdown 摘要复制。

第二期目标不是做机构画像，而是进一步解释研究方向之间如何展开、连接、增长和分化，并帮助用户发现值得持续追踪的研究问题、论文成果和潜在研究空白。

## 5. 非目标范围

第一期和第二期均不做：

1. 不做全文论文精读；
2. 不做自动生成完整文献综述；
3. 不做 AI 聊天式研究助手；
4. 不做个人账号推荐系统；
5. 不做投稿预测；
6. 不做论文质量主观评分；
7. 不做付费数据库深度集成；
8. 不做 Google Scholar 大规模爬取；
9. 不承诺全网绝对完整覆盖；
10. 不把机构排名作为主产品目标；
11. 不以学校合作网络作为主要分析对象。

## 6. 核心用户

### 6.4 导师 / 研究员 / 实验室成员

1. 追踪 HCAI 研究方向的变化；

## 7. HCAI 范围定义

本产品中的 HCAI 指广义的人本 AI / 人智交互研究：

研究人与 AI 系统之间交互、协作、信任、控制、解释、体验、社会影响和认知影响的论文。

判断标准：

1. 如果论文主要研究人如何使用、理解、信任、控制、协作或受到 AI 影响，进入看板；
2. 如果论文主要是模型性能、算法优化或 benchmark，且没有人类用户、交互、体验、行为、心理或社会因素，不进入主看板；
3. 如果论文属于教育、医疗、心理健康等应用方向，但研究重点是用户交互、接受、理解、信任或使用过程，可以进入看板。

每个方向包含：

## 10. 数据源设计

### 10.1 主数据源

第一期优先使用开放学术数据库：

1. OpenAlex；
2. arXiv；
3. Semantic Scholar；
4. Crossref。

各数据源作用：

| 数据源 | 主要用途 |
|---|---|
| OpenAlex | 获取论文、作者、机构、主题、开放获取链接 |
| arXiv | 获取最新预印本，尤其是 HCI、AI、LLM、Agent 方向 |
| Semantic Scholar | 补充引用数、摘要、作者、相关论文 |
| Crossref | 补充 DOI、出版信息、期刊 / 会议、出版商 |

### 10.2 补充源

第二期可增加：

1. 会议官网；
2. 学校实验室 publication 页面；
3. 项目主页；
4. PDF；
5. accepted paper 列表。

补充源只用于回填，不作为主数据来源。

### 10.3 网页搜索兜底

网页搜索只做补充，不作为核心数据源。

适合用于：

1. 查找论文项目主页；
2. 查找 PDF；
3. 查找 accepted paper list；
4. 查找实验室新闻；
5. 补充论文页面；
6. 补充数据集或 demo 链接。

## 11. 更新逻辑

### 11.1 更新时间

系统每天北京时间 0:00 自动更新。

页面展示：

1. 最近一次更新时间；
2. 数据窗口；
3. 数据源状态；
4. 当日新增论文数；
5. 新增研究问题数。

### 11.2 数据窗口

默认抓取最近 72 小时新增或更新论文。

原因：

1. 不同数据库更新时间不同；
2. arXiv、OpenAlex、Semantic Scholar、Crossref 同步存在延迟；
3. 72 小时窗口可以减少漏抓；
4. 通过去重避免重复展示。

### 11.3 更新流程

每天 0:00 执行：

1. 启动更新任务；
2. 从主数据源抓取最近 72 小时新增或更新论文；
3. 使用关键词策略召回 HCAI 候选论文；
4. 去重；
5. 识别标题、摘要、作者、机构、日期、来源和链接；
6. 判断是否属于 HCAI；
7. 计算 HCAI 相关性分数；
8. 归入一级研究方向；
9. 提取二级研究问题；
10. 提取研究方法、应用场景、用户群体、AI 系统类型、交互模式、评价指标、成果类型；
11. 识别参与机构；
12. 更新方向统计；
13. 更新研究问题统计；
14. 更新首页、论文列表、方向列表和方向详情；
15. 生成更新日志；
16. 将低置信度论文放入待审池。

## 13. HCAI 相关性判断

### 13.1 分数范围

每篇候选论文获得一个 HCAI 相关性分数：

| 分数 | 含义 | 展示策略 |
|---|---|---|
| 90–100 | 明确属于 HCAI | 高相关论文 |
| 70–89 | 高度相关 | 进入主看板 |
| 50–69 | 边缘相关 | 进入待审池 |
| 0–49 | 不相关或弱相关 | 不进入主看板 |

第一期默认只展示 70 分及以上论文。

首页“最新高相关论文”展示 HCAI ≥ 85 的论文。

### 13.2 打分维度

HCAI 分数由以下因素综合计算：

1. 是否研究人与 AI 系统的交互；
2. 是否包含用户研究、用户实验、访谈、问卷、田野研究或日志分析；
3. 是否研究信任、依赖、控制、解释、认知负荷、体验、责任或接受；
4. 是否关注 AI 系统对人的行为、决策、态度或认知影响；
5. 是否属于 HCI / HAI / human factors 相关会议或期刊；
6. 是否能提取明确研究问题；
7. 是否只是纯算法、纯模型或 benchmark。

## 14. 内容字段体系

每篇进入看板的论文需要结构化为以下字段。

### 14.1 论文基础字段

| 字段 | 说明 |
|---|---|
| paper_id | 系统内部论文 ID |
| title | 标题 |
| abstract | 摘要 |
| authors | 作者列表 |
| publication_date | 发表或预发表日期 |
| venue | 会议 / 期刊 / 平台 |
| source | 数据来源 |
| doi | DOI |
| arxiv_id | arXiv ID |
| url | 论文链接 |
| pdf_url | PDF 链接 |
| project_url | 项目主页 |
| citation_count | 引用数 |
| open_access | 是否开放获取 |

### 14.2 HCAI 分类字段

| 字段 | 说明 |
|---|---|
| hcai_score | HCAI 相关性分数 |
| primary_direction | 一级研究方向 |
| secondary_directions | 其他相关方向 |
| research_questions | 二级研究问题 |
| classification_reason | 分类理由 |
| review_status | 自动通过 / 待审 / 已排除 |
| confidence | 分类置信度 |

### 14.3 内容分析字段

| 字段 | 说明 |
|---|---|
| research_methods | 研究方法，如用户实验、访谈、问卷、日志分析、田野研究、日记研究、原型系统 |

## 15. 去重逻辑

同一篇论文可能同时出现在 arXiv、OpenAlex、Crossref、Semantic Scholar 和学校官网中。

去重优先级：

1. DOI；
2. arXiv ID；
3. Semantic Scholar Paper ID；
4. OpenAlex Work ID；
5. 标题相似度；
6. 作者列表相似度；
7. 发表年份；
8. 会议 / 期刊信息。

版本合并规则：

1. arXiv 预印本、正式会议版、期刊扩展版、项目页默认合并为一条主记录；
2. 论文详情页展示多个链接；
3. 正式发表信息出现时更新原记录，不重复计数；
4. 低置信度重复进入待审池。

## 26. 指标口径

### 26.1 今日新增

最近一次更新中首次进入系统的论文数量。

如果论文之前已存在，只更新引用数、链接或来源信息，不计入今日新增。

### 26.2 近 7 日 / 近 30 日 / 近 1 年

默认以论文首次进入看板时间统计。

若首次进入时间缺失，则使用 publication_date。

### 26.3 覆盖研究方向

当前系统维护并可展示的一级研究方向数量。

第一期为 18。

### 26.4 新增研究问题

近 30 日内首次出现，或从低频进入可展示阈值的二级研究问题数量。

### 26.5 方向论文数

某方向下进入看板的论文数。

一篇论文可归入多个方向。

第一期统计规则：

1. 主方向 +1；
2. 副方向也可用于方向共现；
3. 方向列表默认按主方向论文数排序；
4. 关系图使用主方向与副方向共现计算。

### 26.6 研究问题数

某方向下可展示的二级研究问题数量。

### 26.7 增长率

增长率 = 本周期论文数 / 上周期论文数 - 1

若上周期为 0，需要标记为“新兴”或使用特殊展示，不直接显示无限大。

### 26.8 HCAI 平均分

方向下所有进入看板论文的 HCAI 分数平均值。

### 26.9 方向共现

如果一篇论文同时归入两个或多个方向，则这些方向之间形成共现关系。

每篇论文中任意两个方向组合的共现次数 +1。

## 32. 数据异常处理

### 32.1 摘要缺失

如果摘要缺失：

1. 仍可进入候选池；
2. 使用标题、关键词、会议和来源判断；
3. 标记为“摘要缺失”；
4. 降低分类置信度；
5. 必要时进入待审池。

### 32.2 方向不确定

如果方向分类不确定：

1. 标记为低置信度；
2. 进入待审池；
3. 不进入方向趋势统计；
4. 人工确认后再计入。

### 32.3 内容标签缺失

如果方法、场景、用户群体无法提取：

1. 允许论文进入看板；
2. 标记缺失字段；
3. 降低数据质量分；
4. 后续通过补充摘要或人工审核完善。

### 32.4 重复论文

如果疑似重复：

1. 自动合并高置信度重复；
2. 低置信度重复进入待审池；
3. 合并后不重复计数；
4. 保留不同版本链接。

## 36. 风险与应对

### 36.1 风险：HCAI 边界模糊

应对：

1. 使用 HCAI 分数；
2. 展示分类理由；
3. 低置信度进入待审池；
4. 分类规则中加入纯算法负向判断。

### 36.2 风险：研究方向归类错误

应对：

1. 维护方向定义和别名；
2. 一篇论文允许多个方向；
3. 主方向用于排序，副方向用于共现；
4. 后台允许人工修正。

### 36.3 风险：研究问题提取不稳定

应对：

1. 建立二级研究问题词表；
2. 同义问题归并；
3. 新问题先进入待确认状态；
4. 新增研究问题需达到展示阈值。

### 36.4 风险：数据更新延迟

应对：

1. 使用 72 小时窗口；
2. 展示最近更新时间；
3. 展示数据源状态；
4. 不承诺绝对实时。

### 36.5 风险：机构信息与方向主线冲突

应对：

1. 机构信息作为辅助字段；
2. 首页和主导航不以高校为中心；
3. 周报和方向画像中可展示活跃机构；
4. 不再用机构榜单定义产品核心。

### 36.6 风险：内容标签提取不准

应对：

1. 保留原始摘要；
2. 显示置信度；
3. 允许人工修改；
4. 低置信度标签不进入趋势统计；
5. 后台积累修正规则。

## 38. 关键产品原则

### 38.1 以方向和研究问题为核心

产品核心不是“高校发了多少论文”，而是“这个领域正在研究什么问题”。

### 38.2 论文必须内容化

每篇论文不仅要有标题和摘要，还要被结构化为研究问题、方法、场景、用户、AI 系统、指标和成果类型。

### 38.3 分类必须可解释

用户需要知道为什么一篇论文属于 HCAI，为什么属于某个方向，为什么被归入某些研究问题。

### 38.4 不追求绝对完整

第一期目标是稳定、可解释、可持续，而不是声称覆盖全网全部论文。

### 38.5 先论文与方向，后关系与趋势

第一期先把论文、方向、研究问题看清楚。  
第二期再做方向关系、趋势、对比和每周摘要。

### 38.6 面向研究切入，而不是排名

产品最终服务的是用户理解研究问题、选择研究方向、发现论文成果之间的关系，而不是比较谁发得多。
